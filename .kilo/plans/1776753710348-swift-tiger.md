## Objective

Restore `scrape_marketplace_sellers.ts` so seller bulk crawling succeeds instead of failing on every seller with `ReferenceError: extractMarketplaceSellerLocationText is not defined`.

## Findings

1. The immediate failure occurs in `src/parsers/dom/marketplace_dom_parser.ts:374` inside `parseMarketplaceSellerFromDom()`.
2. `page.evaluate((id) => { ... })` runs in the browser page context, but the callback references `extractMarketplaceSellerLocationText(sellerContext, name)`.
3. `extractMarketplaceSellerLocationText()` is defined in the module at `src/parsers/dom/marketplace_dom_parser.ts:248`, not in the browser context, so it is unavailable during `page.evaluate` execution.
4. This explains the exact runtime error from the seller bulk run: `ReferenceError: extractMarketplaceSellerLocationText is not defined`.
5. Existing tests cover `extractMarketplaceSellerLocationText()` directly in `tests/marketplace_dom_parser.test.ts`, but there is no test that exercises the `page.evaluate` integration path in `parseMarketplaceSellerFromDom()`.

## Likely Root Cause

The seller DOM parser mixes two execution environments:

- Node/TypeScript module scope: helper functions like `extractMarketplaceSellerLocationText`
- Browser page scope: the callback body passed to `page.evaluate`

The current implementation assumes module helpers are callable inside the browser callback, which Puppeteer does not support unless the logic is inlined or passed as serializable data.

## Fix Plan

1. Refactor `parseMarketplaceSellerFromDom()` to avoid calling module-scope helpers inside `page.evaluate`.
   Preferred approach:
   - Keep `page.evaluate` limited to extracting plain DOM snapshot data only.
   - Return `spans`, `sellerContext`, `name`, `memberSince`, `responsive`, `rating`, `reviewCount`, and listing card data from the browser context.
   - Compute `location` by calling `extractMarketplaceSellerLocationText()` after `page.evaluate` returns, in normal TypeScript/Node scope.

2. Preserve current seller parsing behavior while moving only the environment-sensitive logic.
   Specifically keep:
   - current `memberIndex`, `name`, `memberSince`, `responsive`, and `sellerContext` extraction rules
   - current rating/review parsing behavior
   - current listing card extraction and `normalizeMarketplaceSellerListingCards()` flow

3. Add or extend tests in `tests/marketplace_dom_parser.test.ts`.
   Add focused coverage for:
   - a seller DOM parsing path where seller context contains a valid location string
   - the helper being applied outside browser scope by testing the post-evaluate normalization path if practical
   - a regression case ensuring seller parsing does not throw when location extraction is needed

4. Run targeted verification.
   Minimum verification:
   - run the marketplace DOM parser test file
   - run the seller scraper against one known seller ID that previously failed
   - confirm `output/marketplace-sellers-bulk/<sellerId>/marketplace_seller.json` is written
   - confirm no `ReferenceError` is logged

5. Run a short bulk sanity check.
   Suggested command after the fix:
   - run `scrape_marketplace_sellers.ts` with a very small limit, such as `--limit 3`
   - verify successful completion count and inspect one JSON output

## Implementation Notes

1. Prefer the smallest safe change.
   The fix should stay inside `src/parsers/dom/marketplace_dom_parser.ts` unless testing requires small supporting changes.

2. Do not duplicate the location heuristic in two places unless necessary.
   Best outcome is to reuse `extractMarketplaceSellerLocationText()` from Node scope after the browser snapshot is returned.

3. If the `page.evaluate` return object grows, keep it to plain serializable values only.

4. Avoid changing seller normalizer or extractor logic unless verification shows a second bug after the scoping bug is fixed.

## Risks To Check During Implementation

1. Moving location extraction outside `page.evaluate` must not change the ordering/content of `sellerContext`.
2. Rating and review extraction currently happen inside `page.evaluate`; if those stay there, ensure the returned values remain identical.
3. Seller pages with no `Joined Facebook in ...` text should still fall back to the existing `spans.slice(0, 20)` behavior.
4. If a second seller parser issue appears after this fix, treat it separately instead of widening this change prematurely.

## Expected Outcome

After implementation:

1. `scrape_marketplace_sellers.ts` should stop failing immediately on every seller.
2. Seller profile outputs should be written again for successful runs.
3. The DOM parser test suite should include regression coverage for this evaluate-scope bug.
