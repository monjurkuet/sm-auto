# Marketplace Timestamp And Time Metadata Plan

## Why timestamps are currently incomplete

The Marketplace surfaces already carry a coarse scrape timestamp in application output:

- `MarketplaceSearchResult.scrapedAt`
- `MarketplaceListingResult.scrapedAt`
- `MarketplaceSellerResult.scrapedAt`

But persistence is inconsistent compared with the Facebook posts pipeline:

- durable tables (`marketplace_listings`, `marketplace_sellers`) use DB `now()` for `last_seen_at` and `last_scraped_at`
- scrape tables (`marketplace_search_scrapes`, `marketplace_listing_scrapes`, `marketplace_seller_scrapes`) do not store an explicit observed/scraped timestamp field of their own
- search result snapshots (`marketplace_search_results`) do not store when the listing was observed in the search run
- child rows (`marketplace_listing_images`, `marketplace_listing_delivery_options`, `marketplace_seller_scrape_listings`) also have no time metadata
- views rely on `scrape_runs.completed_at` as a proxy for scrape time

This means the system can answer “when did this run finish?” but not consistently answer:

- when the surface payload itself was observed
- when a specific search result row was seen in the feed
- when a listing image or delivery option was last refreshed
- when seller inventory membership was observed in the seller scrape

- seller inventory join table stores only `(scrape_run_id, position, listing_id)` with no observation time

## Design goals

1. Make Marketplace timestamp handling consistent with page/page-post persistence.
2. Preserve the extractor-produced scrape time instead of replacing it with DB `now()` where semantic “observation time” matters.
3. Keep `scrape_runs.started_at/completed_at` as orchestration timestamps.
4. Add row-level observation timestamps where snapshots are stored.
5. Avoid over-modeling unavailable source timestamps. Only store source-native timestamps if Facebook actually exposes them.

## Recommended timestamp model

Use three levels of time metadata:

### 1. Run lifecycle time

Keep existing:

- `scrape_runs.started_at`
- `scrape_runs.completed_at`

Purpose:

- scheduler/runtime timing
- wall-clock run duration
- failure auditing

### 2. Surface observation time

Persist extractor-level `scrapedAt` on Marketplace scrape tables:

- `marketplace_search_scrapes.scraped_at`
- `marketplace_listing_scrapes.scraped_at`
- `marketplace_seller_scrapes.scraped_at`

Purpose:

- when that structured surface payload was observed
- stable reference independent of transaction timing

### 3. Entity/snapshot observation time

Persist row-level observation time for snapshot/child rows:

- `marketplace_search_results.observed_at`
- `marketplace_seller_scrape_listings.observed_at`
- `marketplace_listing_images.first_seen_at`, `last_seen_at`
- `marketplace_listing_delivery_options.first_seen_at`, `last_seen_at`

And make durable tables use extractor time when available:

- `marketplace_sellers.last_scraped_at = result.data.scrapedAt` rather than `now()`
- `marketplace_listings.last_scraped_at = listing/result scrape timestamp` rather than `now()`
- keep `first_seen_at` / `last_seen_at` as DB observation timestamps unless we decide to align `last_seen_at` to extractor time too

## Scope decisions

### Store source-native listing creation time?

Not in this phase by default.

Reason:

- current Marketplace extractors/parsers do not reliably extract listing posted/created timestamps
- mixing inferred DOM text parsing with missing structured support adds more risk than value

Plan:

- add infrastructure for scraper-observed timestamps now
- optionally add source-native fields later if Facebook payloads expose them reliably

### Reuse `scrapedAt` everywhere or introduce `observedAt`?

Recommendation:

- keep `scrapedAt` in output contracts for backward compatibility
- use `scraped_at` for scrape-table rows
- use `observed_at` for per-snapshot rows like search results and seller inventory membership

This mirrors the page posts pipeline, where `facebook_post_scrapes.observed_at` already exists.

### 2. `upsertMarketplaceListing`

Current behavior:

- uses `now()` for `last_seen_at` and `last_scraped_at`

Planned change:

### 3. `replaceMarketplaceListingImages`

Current behavior:

- hard delete then reinsert, which destroys temporal continuity

Recommended improvement:

- replace full delete/reinsert with upsert + stale deactivation pattern only if image history matters strongly
- minimal viable timestamp plan:
  - keep delete/reinsert for now
  - populate `first_seen_at` and `last_seen_at` on insert
- better long-term plan:
  - stop deleting all rows
  - upsert current image URLs with `last_seen_at = scrapedAt`
  - optionally add `is_active` column if image disappearance needs tracking

Tradeoff:

- minimal plan is easy but loses continuity across runs
- full plan is better historically, but larger refactor

Recommendation:

- phase 1: add timestamps with existing semantics
- phase 2: convert to non-destructive image/delivery tracking if needed

Same tradeoff as images.

Recommendation:

- same phased approach as above

## Risks and mitigations

### Risk: ambiguity between DB `now()` and extractor `scrapedAt`

Mitigation:

- define semantics clearly in docs and code comments
- use extractor `scrapedAt` for scrape/observation fields
- retain DB `now()` for operational fields like `last_seen_at` and run completion

### Risk: migration impact on existing views and downstream SQL

Mitigation:

- additive migration first
- backfill before setting `NOT NULL`
- recreate views in same migration or following migration

### Risk: over-expanding public contracts

Mitigation:

- keep contract changes minimal in first phase
- focus DB schema first

## Final recommendation

Yes, Marketplace timestamps should be stored more explicitly.

The best approach is not just “add one timestamp column”, but to distinguish:

- run lifecycle time
- surface scrape time
- per-row observation time

Implement the minimal timestamp set first, centered on Marketplace scrape tables and snapshot rows, while propagating extractor `scrapedAt` into durable listing/seller `last_scraped_at`. That gives immediate analytical value and aligns Marketplace persistence with the stronger timestamp model already used by page posts.

## Bulk Crawl Plan: Uncrawled Marketplace Listings And Sellers

## Goal

Add first-class CLI commands to crawl all uncrawled Marketplace listing details and seller details using the same style as the existing search command, for example:

- `bun run src/cli/scrape_marketplace_listings.ts --source-query "iphone" --source-location "Dhaka"`
- `bun run src/cli/scrape_marketplace_sellers.ts --source-query "iphone" --source-location "Dhaka"`

And also support a broader mode with no query/location filter:

- `bun run src/cli/scrape_marketplace_listings.ts --uncrawled-only`
- `bun run src/cli/scrape_marketplace_sellers.ts --uncrawled-only`

## Why this should be a separate bulk CLI instead of overloading search/listing/seller CLIs

The current CLIs are single-entity or single-search entrypoints:

- `scrape_marketplace_search.ts` = one search surface
- `scrape_marketplace_listing.ts` = one listing ID
- `scrape_marketplace_seller.ts` = one seller ID

Bulk crawling needs different behavior:

- fetch target IDs from DB
- support queue filtering and limits
- iterate many entities in sequence
- survive partial failures
- produce aggregate output rather than single-entity output

That is a different workflow and should be its own command family.

## Recommended command design

### 1. Bulk listing crawler

Add a new CLI:

- `src/cli/scrape_marketplace_listings.ts`

Primary usage:

```bash
bun run src/cli/scrape_marketplace_listings.ts --uncrawled-only
```

Optional query/location-scoped usage:

```bash
bun run src/cli/scrape_marketplace_listings.ts --source-query "iphone" --source-location "Dhaka" --uncrawled-only
```

### 2. Bulk seller crawler

Add a new CLI:

- `src/cli/scrape_marketplace_sellers.ts`

Primary usage:

```bash
bun run src/cli/scrape_marketplace_sellers.ts --uncrawled-only
```

Optional query/location-scoped usage:

```bash
bun run src/cli/scrape_marketplace_sellers.ts --source-query "iphone" --source-location "Dhaka" --uncrawled-only
```

## CLI options

Both bulk CLIs should support the existing shared options from `parseSharedOptions`, plus these bulk-specific options.

### Shared bulk options

- `--uncrawled-only`
  - boolean
  - default: `true`
  - fetch only entities with no corresponding detail scrape yet

- `--limit <n>`
  - maximum number of entities to process in this run

- `--offset <n>`
  - useful for chunking manual backfills

- `--batch-size <n>`
  - how many IDs to fetch from DB per selection window
  - primarily for internal loop management, not parallel browser execution

- `--dry-run`
  - print the IDs/count that would be processed, but do not scrape

- `--continue-on-error`
  - default `true`
  - do not abort the whole bulk run because one listing/seller failed

- `--delay-ms <n>`
  - delay between entities to reduce Facebook rate-pressure

- `--source-query <query>`
  - restrict candidates to entities discovered by a given search query

- `--source-location <location>`
  - restrict candidates to entities discovered by a given search location

### Listing-specific options

- `--include-recrawls-before <timestamp>`
  - include listings last scraped before a cutoff, not only null/unseen ones

### Seller-specific options

- `--require-listing-history`
  - only crawl sellers that are referenced by at least one marketplace listing row

- `--include-recrawls-before <timestamp>`
  - same pattern as listings

## Selection logic

## Definition of “uncrawled” for listings

Recommended rule:

- listing exists in `scraper.marketplace_listings`
- no row exists in `scraper.marketplace_listing_scrapes` for that `listing_id`

Base selector:

```sql
SELECT ml.listing_id
FROM scraper.marketplace_listings ml
LEFT JOIN scraper.marketplace_listing_scrapes mls ON mls.listing_id = ml.listing_id
WHERE mls.listing_id IS NULL
ORDER BY ml.last_seen_at DESC NULLS LAST, ml.listing_id
LIMIT $limit OFFSET $offset;
```

Scoped-to-search selector:

```sql
SELECT DISTINCT ml.listing_id
FROM scraper.marketplace_search_scrapes mss
JOIN scraper.marketplace_search_results msr ON msr.scrape_run_id = mss.scrape_run_id
JOIN scraper.marketplace_listings ml ON ml.listing_id = msr.listing_id
LEFT JOIN scraper.marketplace_listing_scrapes mls ON mls.listing_id = ml.listing_id
WHERE mss.query = $1
  AND mss.location_text = $2
  AND mls.listing_id IS NULL
ORDER BY ml.last_seen_at DESC NULLS LAST, ml.listing_id
LIMIT $3 OFFSET $4;
```

## Definition of “uncrawled” for sellers

Recommended rule:

- seller exists in `scraper.marketplace_sellers`
- no row exists in `scraper.marketplace_seller_scrapes` for that `seller_id`

Base selector:

```sql
SELECT ms.seller_id
FROM scraper.marketplace_sellers ms
LEFT JOIN scraper.marketplace_seller_scrapes mss ON mss.seller_id = ms.seller_id
WHERE mss.seller_id IS NULL
ORDER BY ms.last_seen_at DESC NULLS LAST, ms.seller_id
LIMIT $limit OFFSET $offset;
```

Scoped-to-search selector:

```sql
SELECT DISTINCT ms.seller_id
FROM scraper.marketplace_search_scrapes mssrc
JOIN scraper.marketplace_search_results msr ON msr.scrape_run_id = mssrc.scrape_run_id
JOIN scraper.marketplace_sellers ms ON ms.seller_id = msr.seller_id
LEFT JOIN scraper.marketplace_seller_scrapes mss ON mss.seller_id = ms.seller_id
WHERE mssrc.query = $1
  AND mssrc.location_text = $2
  AND mss.seller_id IS NULL
ORDER BY ms.last_seen_at DESC NULLS LAST, ms.seller_id
LIMIT $3 OFFSET $4;
```

## Bulk run orchestration design

## Recommendation: sequential scraping first, no parallel browser tabs initially

Reason:

- the current scraper architecture is stable for single-entity jobs
- parallel browser attachment would introduce more instability and throttling risk
- correctness and recoverability matter more than throughput for the first version

Process for each bulk command:

1. Parse shared and bulk-specific options.
2. Query DB for candidate IDs.
3. If `--dry-run`, print counts/IDs and exit.
4. For each ID:
   - run the existing extractor directly

## Humanized Delay Plan For Bulk Marketplace Crawlers

## Problem

The current bulk commands:

- `src/cli/scrape_marketplace_listings.ts`
- `src/cli/scrape_marketplace_sellers.ts`

only support a fixed `--delay-ms` pause between entities.

That creates a machine-like pattern:

- same pause after every entity
- no longer pauses after bursts of activity
- no short jitter before starting a new entity
- no distinction between normal pacing and cool-down pacing after errors

This is easy to spot as automation behavior and is not ideal for long Marketplace crawl runs.

## Goal

Replace the fixed-delay-only model with a humanized pacing system that:

1. varies the delay between entities
2. adds occasional longer pauses
3. keeps behavior deterministic enough to debug
4. does not overcomplicate the CLI or execution flow
5. preserves the current sequential bulk architecture

## Design principles

### 1. Humanized pacing should be a policy layer, not embedded in scraper logic

The pacing behavior belongs in:

- `src/cli/marketplace_bulk.ts`

not in:

- individual extractors
- queue selectors
- persistence code

Reason:

- pacing is orchestration behavior
- extractors should remain focused on one entity scrape
- easier to reuse for both listings and sellers

### 2. Support both fixed and humanized modes

Recommended modes:

- `fixed`
- `humanized`
- `off`

Reason:

- `fixed` is useful for reproducibility and debugging
- `humanized` is the default production-like mode
- `off` is useful for local testing or extremely short runs

## Recommended CLI design

Add new options to both bulk commands.

### Core pacing options

- `--delay-mode <fixed|humanized|off>`
  - default: `humanized`

- `--delay-ms <n>`
  - keep existing flag
  - in `fixed` mode: exact pause
  - in `humanized` mode: base delay

### Humanized delay tuning options

- `--delay-jitter-ms <n>`
  - default: `1500`
  - adds random spread around the base delay

- `--pause-every-min <n>`
  - default: `4`

- `--pause-every-max <n>`
  - default: `9`
  - after a random number of entities in this range, insert a longer rest

- `--pause-min-ms <n>`
  - default: `8000`

- `--pause-max-ms <n>`
  - default: `25000`

- `--error-delay-multiplier <n>`
  - default: `1.75`
  - after a failed entity scrape, next pause is longer

### Optional advanced option

- `--seed <n>`
  - optional deterministic random seed for reproducible behavior during debugging

## Recommended behavior model

Use a pacing planner that decides the next wait interval before moving to the next entity.

### Base delay

For each successful entity in `humanized` mode:

- start from `delay-ms`
- add random jitter in the range:
  - `[0, delay-jitter-ms]` or
  - symmetric jitter around base if you want centered behavior

Recommended implementation:

- `nextDelay = delayMs + randomInt(0, delayJitterMs)`

Reason:

- simpler to reason about
- avoids unrealistically fast negative jitter

### Burst pause behavior

Track a counter of completed entities since the last longer pause.

At the start of the run, sample a random threshold:

- between `pause-every-min` and `pause-every-max`

When the threshold is reached:

- apply a longer delay sampled from `pause-min-ms` to `pause-max-ms`
- reset the counter
- sample a new threshold

This produces a more human rhythm:

- several actions in a row
- then a break
- then continue

### Error-aware backoff

After a failed entity scrape:

- multiply the next standard delay by `error-delay-multiplier`
- optionally clamp to a safe max

Reason:

- if the site is getting unstable or rate-limited, this naturally slows down

### Optional first-action jitter

Before the first entity scrape in a run:

- add a small startup jitter
  - for example `500-2500 ms`

Reason:

- avoids immediate action at process start
- gives more natural pacing

### New helper module

Add:

- `src/cli/humanized_delay.ts`

Suggested exports:

- `createDelayPlanner(options)`
- `sleep(ms)`
- `sampleNextDelay(state, outcome)`

Suggested types:

- `DelayMode = 'off' | 'fixed' | 'humanized'`
- `DelayPlannerOptions`
- `DelayPlannerState`

### Extend `MarketplaceBulkOptions`

Add fields to `src/cli/marketplace_bulk.ts` options type:

- `delayMode`
- `delayJitterMs`
- `pauseEveryMin`
- `pauseEveryMax`
- `pauseMinMs`
- `pauseMaxMs`
- `errorDelayMultiplier`
- optional `seed`

### Update both CLIs

Add these flags to:

- `src/cli/scrape_marketplace_listings.ts`
- `src/cli/scrape_marketplace_sellers.ts`

Then pass them through to `runMarketplaceBulkCommand`.

### Update bulk runner flow

In `src/cli/marketplace_bulk.ts`:

1. initialize delay planner once at run start
2. before first item, apply startup jitter if enabled
3. after each entity:
   - determine outcome: `success` or `failure`
   - ask planner for next delay
   - log chosen delay reason
   - sleep that duration

## Logging and observability

This is important so humanized behavior is explainable.

For each non-zero delay, log something like:

```text
[INFO] Waiting 6421ms before next listing (mode=humanized, reason=normal-jitter)
[INFO] Waiting 18422ms before next seller (mode=humanized, reason=burst-pause)
[INFO] Waiting 10900ms before next listing (mode=humanized, reason=error-backoff)
```

Also add summary fields into the bulk summary JSON:

- `delayMode`
- `baseDelayMs`
- `delayJitterMs`
- `burstPauseCount`
- `errorBackoffCount`
- `totalDelayMs`

This makes runs auditable and tunable.

## Safe default values

Recommended defaults for first rollout:

- `delay-mode=humanized`
- `delay-ms=2500`
- `delay-jitter-ms=1500`
- `pause-every-min=4`
- `pause-every-max=9`
- `pause-min-ms=8000`
- `pause-max-ms=25000`
- `error-delay-multiplier=1.75`

This produces:

- most entity-to-entity pauses around 2.5s to 4s
- occasional breaks around 8s to 25s
- slower recovery after failures

That is still efficient, but less obviously robotic.

## Non-goals for first version

Do not add these yet:

- random mouse movement simulation
- keyboard delays inside extractors
- randomized scroll behavior per page in bulk runner
- proxy rotation logic
- adaptive pacing based on HTTP status codes or anti-bot challenge detection

Those can be added later, but they are not necessary to improve the bulk crawler behavior significantly.

## Edge cases and safeguards

### 1. Very small runs

If only 1 or 2 entities are selected:

- do not force a long burst pause
- only use startup jitter + normal delay behavior if needed

### 2. Dry run

In `--dry-run` mode:

- never sleep
- do not initialize delay planner state beyond config serialization if included in summary

### 3. Fail-fast mode

If `--fail-fast` stops the run immediately after a failure:

- do not apply post-failure sleep after the terminal failure

### 4. Upper bound protection

Clamp any computed delay to a max safe value, for example:

- `maxComputedDelayMs = 60000`

Reason:

- protects against accidental misconfiguration

### 5. Invalid ranges

Validate:

- `pause-every-min <= pause-every-max`
- `pause-min-ms <= pause-max-ms`
- all values are non-negative

Reject invalid CLI combinations early with clear errors.
